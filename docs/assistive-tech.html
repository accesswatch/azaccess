Assistive Technology Coverage

Updated: 2025-11-20 · Next review: 2026-02-20

Overview

Arizona tests new experiences with a small but representative set of assistive technologies so we catch common barriers before they reach students, employees, or visitors.

This page outlines the minimum coverage we expect for major releases.

Screen readers

Platform Primary tools Typical scenarios
Windows JAWS, NVDA, Fusion Quickstart sites, enterprise apps, Brightspace, research dashboards.
macOS VoiceOver Arizona sites in Safari/Chrome, creative tools, LMS content.
iOS/iPadOS VoiceOver Mobile apps, campus maps, authentication flows, event registration.
Android TalkBack Mobile apps, web views, student tools, vendor portals.

Low vision and magnification

  • Fusion/ZoomText on Windows for magnification with speech.
  • Built-in zoom and high-contrast modes on Windows, macOS, iOS, and Android.
  • Ensure content reflows without horizontal scrolling at 200–400% zoom.

See the WCAG 2.2 highlights for focus visibility and target size requirements.

Alternative input and mobility

  • Voice input (e.g., Dragon, built-in dictation) for command and text entry.
  • Switch access and keyboard-only navigation.
  • Pointer alternatives such as head tracking.

Interactive controls must have clear labels so voice users can say “Click <label>”. See the Web & App hub for patterns.

How to plan coverage for your project

  1. Identify critical user journeys (e.g., apply, register, pay, submit, report).
  2. Choose at least one screen reader/platform pair and one magnification setup that matches your audience.
  3. Schedule manual testing before launch and after major UI changes.
  4. Log findings in your team’s issue tracker and surface high-risk issues to the Accessibility team.
